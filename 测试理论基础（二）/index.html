测试理论基础（二）

测试理论基础（二）

先导

软件开发不存在“银弹”（通用的开发方法），所以需要不断学习和使用。

测试（一）的先导提到了测试的概念，这里再补充一下，软件开发的过程。

通俗的讲，软件过程就是软件生产的一系列关联活动的集合。

最直接最经典就是瀑布模型：需求分析、设计、写代码、测试、debug、交付

常用的是敏捷开发：应对快速变化的需求产生的开发方法论。做增量、迭代的开发。

白盒测试

我对白盒的初步印象就是看代码找Bug。据说这是测试领域最难的部分？

​ 白盒测试是一种测试手段，多用于单元测试阶段，测试对象是函数代码等。追求的目标叫做：逻辑覆盖率。然后根据不同的白盒测试方法，来设计测试用例。高手可以用尽量少的用例达到高覆盖目的。

以某一个if-else语句为例，根据以下覆盖方法设计用例：

  1. 语句覆盖，让每个语句都执行一遍。也就是遍历每个分支。
  2. 判定覆盖，也叫分支覆盖，让每个判定的真假都至少出现一次。
  3. 条件覆盖，每个条件的真假都至少出现一次。注意是每个条件。
  4. 判定条件覆盖，把判定和条件组合起来。用尽量少的用例同时满足俩种覆盖率的技术。
  5. 条件组合覆盖，保证 判定的所有情况都至少出现一次 ，而不单单是真假。
  6. 路径覆盖，保证代码中每条路径都至少执行一次，注意是路径不是语句。将多个判定结果的真假进行排列组合，设计各种可能的用例。路径覆盖号称最强的覆盖法，在时间成本不够的情况下，优先选用路径覆盖是最全面的。

条件覆盖关注的是条件本身的真假（condition），判定覆盖关注的是条件组成的某个分支段的真假（decision）。

条件覆盖颗粒度更细，所以通常需要的设计用例会更多些。

测试用例

说了这么多，给你个需求，怎么写测试用例？

解析需求——划定测试点和测试范围——运用测试方法写用例——标头完善——用例规程

  1. 解析需求，头脑风暴不断发散测试点，模糊的地方要明确
  2. 划定范围，是否选用功非接，是否测试不可见字段，IO是否都要测
  3. 用例写标准些，属性标头要完善，用例标头分为：【需求编号】【用例编号】【测试项目】【测试标题】【重要级别】【预置条件】【输入参数】【执行步骤】【预期输出】【备注】【用例提交人】
  4. 用例规程相当于做流程优化，让执行者可以大幅减少执行时间

双V模型

也就是W模型。

瀑布模型，开发写完测试测，没问题就上线（可以想到测试前期多无聊）。

螺旋模型，快速迭代最喜欢的模型，没有明确的起止时间，一切都是无限循环的深渊。

V模型流程：用户需求-需求分析-概要设计-详细设计-编码实现-单元测试-集成测试-系统测试-验收测试；

双V模型，里面的俩个v ，其实一个是开发流程，一个是测试流程，双V代表两个流程是并行的；
开发：需求分析-概要设计-详细设计-编码实现-模块集成-系统构建-系统安装；
测试：需求测试-概要设计测试-详细设计测试-单元测试-集成测试-系统测试-验收测试；

双V模型的具体执行流程

解析原始需求，制定成需求列表-系统需求规格说明书SRS。

有了SRS，开发着手开发功能，测试开始做前端测试；

  1. ST计划（系统测试计划），主要包括比较大方向上的把控，如测试范围、标准、人力资源任务分配，风险预防等
  2. 有了ST计划，进入ST设计阶段，即系统测试设计阶段，主要是测试的环境，方法，策略，类型，工具，脚本等准备；
  3. ST实现阶段，也就是写用例阶段，成果物为《ST用例》和《ST规程》；
  4. ST执行阶段，要等待其他部门达到要求标准才可以（完成集成测试之后），此步骤主要产出为《ST-BUG报告》和《ST测试报告》；
  5. 验收测试，主要有实际完成和计划对比，ST成果物，建议意见书，最终验收测试结论；

开发做好HLD（概要开发设计），服务端测试就可以开始做IT测试（集成测试阶段）：

  1. IT测试主要测试手段是灰盒测试，测试对象大概率是接口和脚本。服务端测试同学在对HLD进行技术评审后，开始写《IT计划》，主要内容依然是对测试范围等大方向的把控；
  2. IT设计，此阶段服务端测试同学要写好《IT方案》，内容偏重于脚本数据等细节准备；
  3. IT实现，此阶段要写好《IT用例规程》和《IT用例》；
  4. IT执行，这步骤必须要等白盒测试同学做好白盒测试后才可以进行。主要产出为《IT-BUG报告》《IT测试报告》；

开发写好LLD（详细开发设计）后，就可以开始白盒测试，工作步骤和IT测试类似：

  1. UT计划，UT测试，也就是俗称的白盒测试，白盒测试员对LLD进行技术评审后会写好《UT计划》；
  2. UT设计，制定《UT方案》，比如用什么覆盖率方法，具体的数据，调用函数的工具；
  3. UT实现，白盒测试用例，成果物是《UT测试用例》和《UT规程》；
  4. UT执行，等开发完成编码实现，此阶段后会有《UT测试报告》《UT-BUG报告》；

开发完成编码实现后，会进行简单自测（代码走查、同行评审、小组内评审），测试则开始UT执行-IT执行-ST执行，最后产品经理完成验收测试。

如果出现时间紧张情况，就会删掉计划-设计-实现 三个阶段，直接让测试同学对着需求文档写用例。

回归测试

回归有四种方法：【覆盖修改法】【周边影响法】【指标达成法】【完全回归法】

覆盖修改法：开发改什么就回归测试什么；

周边影响法：针对有关联的其他功能进行回归，例如改了登录功能，测试连同注册功能一起测试；

指标达成法：规定了最终的结果的回归，回归过程注重结果，只要结果达成即可停止回归测试，例如下单功能，你就要回归好商品展示，价格获取，代金券使用，第三方支付回调，订单列表等等一系列的功能；

完全回归法：一般在大版本提测，大版本上线前 ，或者大量新功能上线的情况。要进行完整的回归，回归的手段多是自动化为主。

α测试、β测试、灰度测试

α测试

小范围的一批专业用户使用一段时间，看看是否有异常。一般国内都是公司自己小组内，或者研发中心等全体人员随意测试几天。

其实就是内测嘛，所以允许存在bug，bug修复也不用太着急。因为不属于正式上线运营，所以也不用承担相应的责任。

β测试

很多软件都会有beta版本，其实就是用于β测试。

公司发布贝特测试版本，大批内测用户进场使用，若发现问题则报告给客服人员，客服人员审核问题是否严重和紧急后，报告给公司负责人。公司在判断bug的影响范围等上 决定是否要立即修复。修复还是给开发和测试进行修复，然后迭代更新。

相较于α测试，范围更大，偏向于压力测试、全链路测试，一旦出现bug，要优先考虑修复成本和时机。对于多数公司而言，beta版本发行就相当于正式上线运营。

灰度测试

对于小公司而言，α和β测试都会合并，叫做灰度测试，选择几分之一的用户作为目标测试人群。
