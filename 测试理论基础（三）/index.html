测试理论基础（三）

测试理论基础（三）

决定在这篇完成测试理论基础的学习，所以会写长点。

之前提到“β测试更倾向于压测和全链路测试”，这里说下压测和全链路测试，这两者都是为了验证软件的稳定性和可靠性，压测目的是确定系统或软件崩溃的极限，全链路测试目标在于验证整个软件系统的端到端流程。

测试团队的核心在于组织、流程、技术（质量铁三角）。测试团队也分等级（CMM5级图），如下是对应等级的特点：

  1. 初始级，产品质量不可预测，测试排期每次都排不准，线上经常出问题，测试流程混乱，缺乏控制；
  2. 可重复级，之前的宝贵经验可重复使用，整个风险控制得到改善；
  3. 已定义级，测试过程可以被描述的很准确清晰，新人可以迅速理解并加入工作；
  4. 已管理级，整个测试过程可以被测量和量化统计，并且有抓手控制全程；
  5. 优化级，全团队向流程优化迈进，不断的改进测试工具和流程，测试效率不断提高；

作为了解就好，不是主要内容。

测试领域的二八原则：

  1. 20%的功能占据了80%的总bug数
  2. 20%的测试排期测出了最终的80%的bug
  3. 20%的需求需要耗费80%的测试排期
  4. 20%的用例做成自动化是最舒服的

以后做事可能会用到这个规律。

梳理需求

测试开发如何向产品经理一样梳理需求？

测试开发的主要作用就是生产测试工具软件，但因为需求不明所以会开发一些无用功能。所以了解下产品经理梳理需求的大致流程：

  1. 什么情况下需要需求管理？需求对齐、新需求、业务变化、技术升级等；
  2. 需求开发，这个阶段就是双V模型中需求分析的结果，形成《SRS》，然后拿去和用户确认；
  3. 需求管理：人员分配、评审、制定标准、需求跟踪；
  4. 需求跟踪的意义：跟进度，利用矩阵，跟踪需求关联到的开发和测试同学，关联到的用例编号，关联到的bug编号等等。

Bug分类

秒答：和需求文档不一样的就是bug。

秒答是在第一层，接下来详细介绍：

  1. 功能性缺陷
  2. 系统缺陷
  3. 加工缺陷
  4. 数据缺陷
  5. 代码缺陷

黑盒测试

终于到正题了。前面那些做个了解就好，其实根据二八法则，测试理论中就20%的知识在实践中时主要用到的（狗头）。黑盒测试有哪些用例设计方法？等价类、判定表、边界值、正交、场景法……

等价类

主要划分就是有效等价类、无效等价类。

有效等价类，对程序软件来说是合理的输入，有意义的，目的是检查程序是否满足了功能需求。

无效等价类，对程序来说不合理的输入数据，目的是检查程序的错误处理等，无效等价类每条用例只能包含一个无效的测试点。

测试类型：弱一般等价类；强一般；弱健壮性；强健壮性；

等价类测试原则：细化范围，根据规则取有效类和无效等价类；

多做题多练练，记太多没用。

判定表

主要是处理多个输入，且多个输入之间有逻辑关系的情况。测试的对象并非输入本身，而是多个输入之间的逻辑关系。

条件桩、动作桩、条件项、动作项，最后组成表格，列出可能的表格。

判定表的分类：实际上，大多数情况下都可以把条件项变成真假 。比如符合不符合，生效不生效等。

列出真假可能，然后把不可能出现的组合划掉。剩下的就是有效的判定表用例了。

现在知道是啥了，剩下就是多练。

边界值

对输入的位于边界的值和稍低稍高的值进行测试。分为离点，内点，上点和特殊点。

边界值用例设计原则：

  1. 输入条件规定范围，则取刚到达这个方位的边界的值和刚超越这个范围边界的值。
  2. 规定个数，则用最大个数，最小个数，最大+1，最小-1。
  3. 根据输出条件使用原则1、2
  4. 如果输入域和输出域是有序的，则去第一个和最后一个元素作为测试数据；
  5. 如果使用了内部数据，则应该用内部数据的边界上的值作为测试数据；
  6. 分析规格说明，找出隐藏的边界条件。

正交

主要作为处理多个输入的测试方法，但是和判定表不同的是，正交处理的多个输入之间不能有任何逻辑关联。

正交只要保证任意条件的子状态和另一个输入条件的任意子状态 都同时出现过 即可。也就是说，它保证任意俩个子状态的排列组合。

场景法

也叫流程图法，是针对一些复杂场景和流程的测试方法，比如取款机。

场景法分为基本流和备用流；

基本流：最主要，最多频率， 最核心，最符合用户使用场景的流。比如：插入银行卡-输入密码-取款-输入金额-确定-取款成功-退卡。

备用流：插入银行卡-输入密码-密码错误-退卡。插入银行卡-输入密码-密码错误-重新输入-成功-退卡。备用流一般会有很多。

每一个流程都是一个用例。

其他方法

规范导出法：根据需求或者说明书，逐句进行验证，保证所有提到的功能都是正常的。

错误猜测法：根据经验，直接测试出错概率最高的功能，可以极大的提高测试性价比。（天天猜）

基于故障测试：目标是要证明 某个特定的故障bug不存在。

基于风险测试：哪有风险就测哪。（例如弱网环境下测试）

比较测试：横向是ios和Android同时测试；纵向则是多个版本一起测试；同行测试；市场占用率高的机型或环境下测试。

怎么谈白盒-黑盒-灰盒

黑盒就是不看代码，白盒就是看代码，灰盒就是接口测试。

幽默回答，可以这么记，但面试回答根本不够看的。

黑盒测试

测试对象是完整的功能或系统；

测试方法：十一种黑盒用例设计；

目的是为了尽早达到验收/上线标准，验证功能和需求是否一致；

介入时间是在功能完成后；

介入条件：无；

评估标准：需求覆盖率；

最低标准：保证必要性，不保证充分性；

优点：简单、模拟用户操作。

缺点：测试介入晚，测试不充分，修复bug代价高。

灰盒测试

测试对象：模块，接口，微服务，脚本等。

测试方法：自下而上，自上而下，大突击，三明治；（芝士什么？）

目的：逐步集成，尽早定位bug。尽早展开测试。

介入时间：内部接口等对象完成开发；

介入条件：出现难定位的bug；关联度复杂度高的接口；软件质量要求极高；跟生命安全相关功能。

评估标准：接口覆盖率

优点：定位准确，介入早于黑盒测试；

缺点：难度大，技术要求高，工作量也大；

白盒测试

测试对象：函数，代码，接口。

测试方法：五种白盒用例设计法；

目的：把bug修复成本和风险降到最低；

介入时间：开发同事写代码过程中；

接入条件：业务，技术，团队达到充分配合和较高技术水准和认知；

评估标准：逻辑覆盖率

最低标准：看以往软件质量定标准

优点：能测到代码内部，发现很多隐藏bug

缺点：难度最高，工作量最大；

使用场景：占工程的20%代码量即可。常用功能，核心功能，复杂度高功能，公共模块，复用度高功能；新研发同事，研发同学技术差，自测不充分的情况。

开展步骤：依据LLD画出函数代码的流程图→变成圈图→整理出代码路径→写成测试用例。

中间件测试

中间件是什么？队列、缓存，都算是。

中间件的特点是：

  1. 可复用性：多次使用，并发使用，不同调用
  2. 可观察性: 从输入和输出来观察这个构建的作用等
  3. 可追踪性：属性状态和行为状态定位问题简单明了
  4. 可控制下：输入处理输出操作行为都可以人为操控，俗称抓手
  5. 可理解性：推广简单，让新人一学就会

测试难点

中间件测试有哪些可能的难点？

  1. 异构性：不同语言、不同平台，在兼容和操作上存在测试挑战
  2. 是否可以看到代码，代码加密就难办
  3. 构建系统的即插即用特性，要求无需编译配置即可升级，版本升级测试就显得更加重要

中间件的测试方法

  1. 通过接口的输入输出来找出中间件的特点
  2. 第一阶段是测试中间件本身，第二阶段要把融合中间件的软件进行测试
  3. 根据各种错误码等规则进行针对性的测试
  4. 给中间件增加一个内部测试环境或者后门程序，出结果，看日志更方便
  5. 找出组件之间的各种交互，然后整理成形式化的测试需求，再写成用例
  6. 一般开源组件基本没有bug，但是仍要进行多个不同组件集成的测试，就像一个工作台进行组装
  7. 防止广告行为，必须找第三方验证

可以看出这7点根本记不住，也没有什么专有名词（某某特性）概括，只能想多少说多少，其实用的时候也是这样的，想到的是怎么做，而不是念着专有名词，至少我是这样的。

自动化测试

学一个最简单的自动化脚本需要多久？答 ：五分钟。

明白自动化测试的意义需要多久？据说要五年。。。

问题：自动化测试脚本你会写，但是你不知道如何写，不知道为什么写，不知道脚本什么时候运行最好，不知道脚本的意义何在。

面试官问谈谈对自动化测试的理解：（以下是新手回答）

  1. 能降低手工测试的执行成本
  2. 用xx语言xx框架可以做
  3. 能回归用例，也能线上轮询监控

自动化测试技术有哪些

  1. 代码的静态动态分析：静态就是不执行源代码，动态就是监控运行时检查代码错误、性能问题；
  2. 测试过程的录制和回放
  3. 脚本技术：编写脚本来模拟用户操作、执行测试用例等。
  4. 虚拟用户技术：模拟大量用户同时使用软件的情况，测试软件在高负载下的性能和稳定性；
  5. 自动管理技术 ：自动化地管理软件测试的各个环节，如测试环境的搭建、测试数据的准备、测试结果的收集和分析等；
  6. 等等……

总结自己自动化项目用了哪些

自动化测试的限制

也就是哪些情况不适合自动化测试：人工测试很容易、

自动化测试的缺点：bug发现的又浅又少；对测试质量依赖性大；成本风险高；软件的部分反自动化设计；工具本身比较木讷。

如何写好脚本

脚本注释、功能、结构（易读，易理解，易维护）、文档

脚本结构

线性脚本，最拉的脚本。

结构化脚本，选择，分支，迭代，循环等都有，好处是可重用和维护简单。

共享脚本：属于组件类脚本，可以同时被多个用例调用，属于很底层的东西，可靠性一定要高，且要做好跟踪日志。

数据驱动脚本：业务逻辑简单，数据多样化的情况适用。

关键字驱动脚本：代码自动生成，用关键字自然语言进行控制，利于推广和维护。

其实说了这么多不如自己去尝试写个自动化脚本

性能测试

如果性能测试能说的很全面，那么offer会非常稳定。

指标

  1. 响应时间，请求全程所耗的时间，是用户最直观的感受，引起响应慢的问题原因非常多；
  2. 吞吐量并发数，单位时间内处理请求的数量，同时承担用户的数量等，经常被用于负载测试；了解一些指标参数的缩写，例如TPS、QPS
  3. 资源利用率，单位时间内系统占用的资源，当然是越低越好；

视角：用户（好用就行）、管理员（所有细节通通关心）、开发人员（找原因、找瓶颈、优化方向）

目标：发现缺陷、性能调优、能力检验与规划。

分类

  1. 性能测试：主要检查是否符合规格说明的需求；
  2. 并发测试：主要检查多个用户同时使用的时候是否会让指标降低；
  3. 压力测试：在规定的大需求压力下能否正常工作；
  4. 可靠性测试：在超规格极限的环境下能否正常使用各个功能；
  5. 负载测试：不断增加业务压力，探索最大指标；
  6. 配置测试：不同环境对性能的影响测试；
  7. 失效恢复测试：容错能力，出现故障后能否恢复；

压测、可靠性、负载，这三者之间的关注点不同。

压测关注点在系统在高负载下的表现，如内存泄漏、处理延迟等。

可靠性关注软件的长期稳定性，包括在连续运行和高负载情况下的表现。

负载测试关注系统在正常负载下的性能指标，如响应时间、吞吐量等。

这三者如何实现？在实践上有什么不同？

Web测试

所谓的web测试，就是b/s架构，就是浏览器页网站的测试。

特点

  1. 基于无连接协议，HTTP/HTTPS是无连接协议；
  2. 内容驱动，文本、图片、视频；
  3. 开发周期短，Web特点
  4. 演化频繁，Web特点
  5. 安全要求高，经常处理敏感信息；
  6. 美观要求高，要关注UI和用户体验；

策略

Web应用分很多层，最好的测试策略就是分层测试。

写过Java前后端项目的应该都知道，表示层属于前端部分，负责页面布局；业务层是接受响应后做出的处理，涉及大量API接口和实现，所以适合接口测试和白盒；数据层是数据库交互的持久层部分，使用框架有MyBatis，写一些增删改查逻辑的，可以拿大量数据测试鲁棒性。

层间测试就是测试这些分层之间逻辑处理是否有缺陷，属于集成测试。

以上是我的理解。

  • 表示层测试：排版结构，链接结构，客户端程序等测试
  • 业务层测试：单个程序白盒测试，一组功能集成测试
  • 数据层测试：破坏性数据测试，大数据测试
  • 层间集成测试：表示层与业务层的集成测试，表示层与数据层，业务层与数据层的测试

测试方法

前面讲测试策略的时候讲到过前后端开发的特点，不难推出需要使用的测试方法。

  • 功能测试，链接测试、表达测试、cookie测试
  • 性能测试，并发、负载、压测、配置、调优
  • 易用性测试，界面UI排版、外观兼容
  • 内容测试，数据库内容是否正常、接口数据的显示等
  • 安全测试，服务器端、客户端、cookie、日志等安全测试
  • 接口测试，接收请求的能力和发出请求的能力测试

安全防护常用测试方法

这是网安的活吧

入侵测试，查看主机日志找到异常、定期检查文件权限、查找网络包找到入侵ip、

漏洞扫描，OS漏洞、网络漏洞、数据库漏洞

安全策略，物理上保护、访问安全（控制权限、网络检查、防火墙）、信息加密、安全管理

兼容性测试

平时怎么做app的兼容性测试时，你是否只会回答，不同手机，不同品牌？不同系统版本？

显然，回答可以看出是否是内行（虽然是外行），但至少要提前了解，回答要有章法。

从三个方面考虑：硬件、软件、数据。

硬件

测试应该先考虑清楚硬件可能造成的影响再选择。

影响：配置和性能的不同、某些特定硬件的功能和设计。

测试内容：与整体硬件的兼容、与外设的兼容。例如，笔记本电脑的数位画板，在mac和windows都可以运行，但是也要考虑到mac的蓝牙键盘 鼠标，和windows的鼠标显示器等等外设是否也能兼容。

软件

  1. 操作系统
  2. 数据库系统
  3. 浏览器
  4. 中间件
  5. 其他软件
  6. 平台软件，包括开发平台和运行平台

数据

  • 不同版本的数据兼容性：比如升级app后 旧数据能否在新接口和客户端完美展现
  • 不同软件间数据兼容性：比如从postman导出的文件能否在接口测试平台导入

功非接

面试官或者你的领导突然给你一个功能，要你设计用例，你要怎么下手？

如果直接从具体的测试用例开始，然后想到多少说多少，往往分数不高。

所以不能忽视条理性，也就是前面提到的章法。目的就是为了能表现自己的思维更全面更系统。

前面提到过，功非接就是功能、非功能、接口。

功能测试

最熟悉的功能测试点。

思考上要从，输入-处理-输出，三个角度讲，并且要分可见和不可见。

非功能测试

可以参考【iso9126】的这个国际质量体系来测试。

其实就是抓住前面提到过的非功能六大特性去讲。

非功能 = 功能性+可靠性+可用性+效率性+可维护性+可移植性

接口测试

不是http协议等接口或者Post/Get，而是更宏观的接口：用户接口，软件接口，硬件接口。

  • 用户接口：交互、表层设计等；
  • 软件接口：软件和其他软件的配合使用情况，比如word文档和pdf互相导入导出的；
  • 硬件接口：软件和其他硬件物理设备的交互质量，比如word文档和打印机，美颜app和摄像头，鼠标驱动和鼠标等；

《理论基础》到这就结束了，有些内容有待实践过程中细化，其主要目的是为了打下基础，所以参考别处博客的时候都是有什么吃什么。

回看这三篇笔记，主要内容还是功非接、白盒、黑盒、自动化、性能。
